# Cross-site request forgery (CSRF) token

A cross-site request forgery (CSRF) token is an unpredictable secret value generated by a web application and tied to a user's session, included in every state-changing request as a defense against CSRF attacks. Because the token is unique and unknown to the attacker, they cannot craft a valid forged request even if they can predict all other parameters.

- **How it works:** When a user loads a page containing a state-changing form or action, the server generates a unique token and embeds it as a hidden field in the form or includes it in a response header. When the form is submitted, the server validates that the token in the request matches the one associated with the user's session, rejecting any request that does not include a valid token.
- **Token properties:** An effective CSRF token must be unpredictable, unique per session or per request, and validated server-side on every state-changing operation. Tokens that are static, predictable, or not validated on the server side provide little to no protection.
- **Common implementation flaws:** In bug bounty hunting, CSRF token implementations are frequently found to be flawed in ways that render them ineffective. Common issues include tokens that are not validated server-side, tokens that remain valid after logout, tokens tied to the user rather than the session, and applications that accept requests with a missing or empty token field.
- **Token placement:** CSRF tokens are typically embedded as hidden input fields in HTML forms, included as custom HTTP request headers in JavaScript-driven applications, or passed as URL parameters, though URL placement is generally discouraged as tokens can appear in server logs and browser history.
- **Double submit cookie pattern:** An alternative implementation where the same random value is sent both as a cookie and as a request parameter. The server validates that both values match, providing CSRF protection without requiring server-side token storage.
- **Relationship with SameSite cookies:** Modern browsers support the SameSite cookie attribute, which restricts when cookies are sent with cross-origin requests and provides a complementary layer of CSRF protection. However, SameSite alone is not universally relied upon as browser support and configuration varies.

CSRF tokens are the primary defense against CSRF attacks, but their effectiveness depends entirely on correct implementation. Flawed token validation is a consistently productive area of investigation in bug bounty programs, as subtle implementation mistakes can render the protection completely ineffective.
